## 一、知識庫：卡片盒筆記法的 AI 原生實現

### 1.1 核心想法

使用者讀到一篇文章時，可以建立一個主題資料夾（如「AI 導入企業」），AI 自動將文章拆解為不同的知識點，放入該資料夾。未來閱讀其他文章時，若其中有相關知識點，也會被自動切出並歸入對應資料夾。

這個做法本質上是「卡片盒筆記法」（Zettelkasten）的 AI 原生版——AI 自動完成卡片拆解、分類與歸檔，進一步還能自動建立卡片之間的連結。

### 1.2 與現有架構的對應

| 概念 | 對應的 Object Model 實現 |
|------|--------------------------|
| 知識點（卡片） | `KnowledgeCard` 或 `Insight` type 的 Object |
| 主題資料夾 | `Topic` 或 `Project` type 的 Object |
| 卡片之間的連結 | `relations` 表的 `relates_to` |
| 卡片盒筆記法 Skill | Personal Skills 中的分類用 Skill |

### 1.3 可切換的分類模組

長遠規劃是提供多種分類方式，使用者可以載入不同的分類模組來客製化。具體做法：

- 每個分類模組本質上是一組規則或 Prompt，決定如何對 Object 進行 grouping 和 tagging。
- 切換模組時，不搬動資料，而是重新跑分類邏輯，生成新的 relations 和 tags。
- 底層需要一個「分類無關」的資料層——Object 本身只存原始內容和 embedding，分類方式是「視圖層」的事。

**風險：** 使用者在 A 模組下手動建立的連結和修正，切到 B 模組後可能消失或衝突。需要「分類層版本控制」——每個模組產生的分類結果獨立存儲，切換時是切換「鏡頭」而不是改動底層資料。

## 二、思考庫：論點與問題的收集場

### 2.1 核心想法

當使用者有了一個論點，可以記錄該論點的正反面敘述，以及遇到的問題。被標示為「問題」的內容會被獨立收集，作為未來的突破口——這些問題可能暫時無法解決，但收集起來後，某天發現解法時就是一個好的切入點。

### 2.2 對應的 ObjectType

| 概念 | 建議的 ObjectType |
|------|-------------------|
| 論點 | `Argument`（含正反面敘述） |
| 待解決問題 | `OpenQuestion` |
| 問題收集區 | filtered view（過濾所有 type 為 `OpenQuestion` 的 Object） |

思考庫本質上是使用者「怎麼想」的容器——AI 可以幫忙收集相關素材，甚至提示矛盾之處，但寫下正反面論述、決定問題是否已解決，這些動作必須由使用者主動觸發。

## 三、最小單位的定義

### 3.1 核心問題

知識庫和思考庫的最小單位要怎麼切？

### 3.2 建議

最小單位不應是固定的一種東西，而是**「帶有語意完整性的原子陳述」**——每一條都能獨立被理解、獨立被引用、獨立被反駁。

- 知識點範例：「RAG 的 recall 與 precision 存在 trade-off」
- 論點範例：「企業導入 AI 應該從流程自動化開始而非知識管理」

### 3.3 切割粒度的兩難

| 切太細（句子級） | 切太粗（段落/文章級） |
|-----------------|---------------------|
| 關聯爆炸、噪音太多 | 失去卡片盒的核心價值——無法重組 |
| 使用者管理不了 | 等同傳統筆記，無法跨文章組合觀點 |

**MVP 建議：** 一個知識點 ≈ 一個可以用 2-3 句話表達的獨立概念。AI 做初步切割，使用者確認或合併。

## 四、原始文章保留策略

### 4.1 雙層結構：原料庫 + 成品庫

保留原始輸入的文章，形成「原料庫」與「成品庫」的雙層架構：

- **原料庫：** 原始文章（`SourceArticle` type），不可變。
- **成品庫：** 根據分類邏輯拆解出的知識卡片，透過 `derived_from` relation 指回原文。

### 4.2 為什麼這比重新處理卡片好

換一個分類模組時，是拿同樣的原料（原文）重新加工，而不是把已切好的卡片重新拼回去再重切。AI 重新讀原文、重新拆解，產生新的卡片，舊卡片可保留或歸檔。

### 4.3 重新分類時的智慧處理

重新分類不是從零開始，而是參考使用者在舊分類下做過的所有修正。例如使用者曾將某張卡片從「AI 導入企業」移到「組織變革」，這個修正行為本身就是有價值的信號，新的分類模組應將其當作輸入之一。

## 五、自動化與學習的平衡

### 5.1 核心原則

要做到最大程度的自動化，但又不能完全自動化。完全自動化會讓使用者失去學習的中心主軸。目標是：讓認知資源專注在學習、判斷和思考上，把與學習無關的雜務自動化。

### 5.2 劃分標準：AI 負責「看到什麼」，使用者負責「怎麼想」

**AI 自動做的事（認知雜務）：**
- 拆解文章成卡片
- 推斷歸屬主題
- 標註出處
- 偵測與已有卡片的潛在關聯並提示

**使用者親自做的事（真正的學習）：**
- 確認分類是否合理
- 寫下自己對知識點的理解或評論
- 決定兩個知識點之間的關係（支持 / 矛盾 / 延伸）
- 將觀察升級為論點
- 將困惑標記為待解決問題

### 5.3 UI 設計啟示

知識卡片的「內容區」由 AI 自動填充（可編輯），但每張卡片下方有一個「我的筆記」區域，永遠是空白的，只有使用者自己能寫，AI 不會填充。這個空白本身就是設計訊號——告訴使用者「這裡是你思考的地方」。

## 六、AI 分類與人工修改的融合

### 6.1 確定性層級

每一筆分類資訊都帶 `source` 標記，分三級：

| 層級 | 說明 | 重新分類時的處理 |
|------|------|-----------------|
| `ai_inferred` | AI 自動產生 | 可直接覆蓋重來 |
| `ai_confirmed` | AI 產生、使用者確認過 | 保留但可提示使用者是否更新 |
| `user_defined` | 使用者自己建立或手動修改 | 絕對不動 |

### 6.2 雙層內容分離

卡片上同時存在 AI 產生的內容和使用者產生的內容時，在資料結構上分開存儲：

- `content`：AI 自動擷取的原文段落
- `user_notes`：使用者的手寫筆記
- `properties` 中每個欄位帶 `_source` 後綴（如 `"topic_source": "user_defined"`）
- `relations` 表加 `source` 欄位，區分 AI 建立的連結和使用者手動建立的連結

### 6.3 效果

切換分類模組重跑時，系統精準知道：哪些可以安全重新產生（AI 層），哪些必須原封不動保留（使用者層）。重跑完後，將使用者層的內容「嫁接」到新的分類結果上。

### 6.4 分類過程中的使用者參與

- AI 信心分數高的分類：自動歸類，事後讓使用者看到並可修正。
- 信心分數低的、或一張卡片可能屬於多個主題的：彈出確認讓使用者選。

避免兩個極端：全自動讓使用者失去掌控感，全手動回到 Obsidian 那種「認知負擔在使用者身上」的老路。

## 七、知識庫與個人系統的切分

### 7.1 兩者的本質差別

| 維度 | 知識庫 | 個人系統 |
|------|--------|---------|
| 內容來源 | 外部（文章、書、對話） | 自己的實踐和反思 |
| 性質 | 描述性——「我知道什麼」 | 指令性——「我怎麼做事」 |
| 內容範例 | 知識點、觀點、事實、別人的想法 | 方法論、思考框架、決策原則 |
| 組織方式 | 卡片盒筆記法分類 | 方法論累積與版本演進 |

### 7.2 兩者之間的流動關係

知識庫是上游的輸入，個人系統是下游的結晶。例如：在知識庫裡讀了十篇關於寫作方法的文章，最終在個人系統裡沉澱出「我的寫作流程：先列論點骨架 → 每個論點找兩個支撐 → 最後寫引言」。

### 7.3 個人系統的用途

1. **累積與演進：** 隨著收集的方法論越來越多，可以將新舊方法在此進行對修與思考。
2. **外部工具與 Agent 協作：** 其他 AI 工具可以存取個人系統作為 context，知道使用者要用什麼樣的方法論去思考或工作。

### 7.4 對應到 Object Model

建議新增 `Methodology` ObjectType：

| 欄位 | 說明 |
|------|------|
| 適用場景 | 寫文章 / 做決策 / 評估技術方案 |
| 步驟流程 | 結構化的方法論步驟 |
| 判斷原則 | 決策時的考量標準 |
| 已知例外 | 方法論不適用的情況 |
| 版本歷史 | 方法論的演化紀錄 |

與知識庫的 relation 類型：`distilled_from`——標記方法論是從哪些知識卡片中提煉出來的。

## 八、個人系統作為 MCP 服務

### 8.1 核心概念

個人系統不需要匯出，而是直接作為 MCP Server 或 CLI，讓外部 AI Agent 即時呼叫。匯出是一次性快照，MCP 是活的連線。

### 8.2 運作方式

任何 Agent 透過 MCP 呼叫 Life Console，查詢「這個使用者評估技術方案的方法論是什麼」，系統直接回傳最新版的 Methodology 物件。Agent 注入自己的 context，用使用者的框架工作。

**與匯出的本質差異：** 使用者昨天更新了方法論裡的某個步驟，今天所有連接的 Agent 自動就用新版。個人系統永遠是 single source of truth。

### 8.3 精細的查詢機制

不是把整個個人系統一口氣倒給 Agent，而是 Agent 根據當前任務查詢相關的方法論。例如在 Claude Code 裡寫程式，Agent 透過 MCP 只問「code review 原則」，不會拿到寫作方法論。

### 8.4 對產品定位的影響

Life Console 不只是「你打開來用」的桌面應用，同時是在背景持續運行的**個人 context server**。使用者日常可能不打開 Life Console 的介面，但所有 AI 工具都在背後連著它。

在 AI Agent 生態中的定位：Life Console 不是又一個 Agent，也不是又一個工具，而是**「個人認知的基礎設施層」**。其他 Agent 和工具是消費者，Life Console 是 provider。

### 8.5 技術可行性

只做唯讀的 MCP endpoint，讓外部 Agent query Methodology 物件，工作量很小——本質上是在 SQLite 上面包一層 MCP Server，提供 `getMethodology(topic)` 和 `listMethodologies()` 兩個 tool。

可考慮提前到 MVP 後期實作，因為這個功能本身就是最好的產品 demo：「裝好 Life Console，然後去 Claude 裡對話，你會發現 Claude 突然懂你了。」

## 九、分階段落地建議

### Phase 1（可立即做）

- 在現有 Object Model 上新增 `KnowledgeCard`、`Argument`、`OpenQuestion`、`SourceArticle` 四個 ObjectType。
- 用 Workflow Plugin 做一個「文章拆解器」，輸入一篇文章，AI 自動拆成多張卡片。
- 用 `derived_from` relation 連結卡片與原文，用 `relates_to` 連結卡片與主題。

### Phase 2

- 建立卡片盒筆記法 Skill，讓 AI 在閱讀新文章時自動判斷知識點歸屬。
- 做「問題收集」的 filtered view。
- 新增 `Methodology` ObjectType，建立個人系統的基礎。
- 實作唯讀 MCP endpoint，讓外部 Agent 查詢 Methodology。

### Phase 3+

- 可切換的分類模組系統。
- 分類層版本控制。
- 個人系統的完整 MCP 服務（含寫入能力）。
